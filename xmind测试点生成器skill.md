# 测试点导图生成器

## 概述

本技能只做一件事：**根据设计稿与需求材料生成测试点导图，并输出 xmind 8 旧版兼容 `.xmind` 文件**。

主输出是“测试点导图”，不是需求分析简报。若用户额外提供需求分析简报，它仅作为辅助输入材料，用于帮助理解规则、范围和风险；**不能替代原始需求，更不能成为主交付物**。

本技能适合复杂交互型产品，尤其是包含以下特征的项目：

*   多页面 / 多步骤流程
*   跨页面参数透传
*   异步任务创建、排队、生成、失败、超时、重试
*   列表状态刷新与结果回流
*   会员 / 积分 / 订阅 / 支付等商业化分流
*   权限、弱网、前后台切换、系统能力依赖
*   再创作 / 二次消费 / 二次编辑

本技能默认采用**轻量结构**：核心规则全部写在本文件中，不依赖额外 `scripts/` 或 `references/`。如运行环境支持代码或文件操作能力，可自行使用合适工具构造 `.xmind` ZIP 包并执行校验；但**本技能本身不要求必须内置脚本**。

## 输入范围

优先读取以下材料：

1.  figma 设计稿
2.  prd / 需求 md / pdf 文档
3.  需求分析简报（可选）
4.  补充澄清记录（可选）
5.  阶段零范围确认结果（可选）

说明：

*   第 1 和第 2 类通常是主证据。
*   第 3 至第 5 类属于补充证据；未提供时不影响执行。
*   若设计稿与需求正文有冲突，优先采用**更明确、更新、可执行**的信息。

## 执行前提：模块名必须手工声明

开始前，必须先从用户请求开头读取本次模块名单。推荐格式：

```text
本次模块为：模块a、模块b、模块c
```

规则如下：

*   以用户本次声明的模块名为准。
*   声明了几个模块，就生成几个 `.xmind` 文件。
*   不允许擅自把文档中的页面列表全部当成模块名单直接生成。
*   若用户没有声明模块名，应先要求补充；不要自行猜测。
*   所有后续的模块归属、自检、文件命名都以这份声明为准。

## 核心目标

1.  基于输入材料，生成**颗粒度可直接继续拆成测试用例**的测试点导图。
2.  导图必须优先体现**业务闭环、状态闭环、数据闭环、权限分流、商业化分流、结果回流、二次操作**。
3.  输出必须是 **xmind 8 / 旧版兼容格式**，且通过结构自检后再交付。

## 证据优先级与冲突处理

当多个输入之间存在差异时，按以下优先级处理：

1.  用户在当前对话中的明确要求
2.  阶段零范围确认结果
3.  补充澄清记录
4.  prd / 需求正文
5.  figma 设计稿与交互稿
6.  需求分析简报

补充原则：

*   需求分析简报只用于辅助理解，不高于原始需求正文。
*   figma 主要补足交互、状态、页面回流、元素启用条件等信息。
*   无法确认时，不要把猜测写进主导图；单独归入“待确认事项”。

## 工作流

按以下顺序执行：

1.  读取用户手工声明的模块名。
2.  通读输入材料，抽取项目名 / 版本号；若无明确版本号，可用“项目名 / 当前版本”。
3.  清洗格式噪音：忽略 html 标签、markdown 噪音、图片链接、表格杂质、实现备注、讨论纪要原文等非正式内容。
4.  建立模块归属：把每条需求、状态、数据流、跳转、回流、权限、商业化分流映射到对应模块。
5.  先为每个模块形成测试点结构草案，再生成最终导图。
6.  直接构造 xmind 8 旧版兼容 `.xmind` 文件。
7.  对每个 `.xmind` 执行结构校验与解压校验；若失败，先修复再交付。

## 导图设计原则

### 1. 不是对称模板，而是链路型测试设计

导图优先采用**横向链路型**写法，不要机械地把每个模块都写成一样。

优先表达以下逻辑：

*   入口来源
*   输入项 / 前置数据
*   提交条件 / 拦截条件
*   中间态 / 任务状态 / 刷新时机
*   结果展示 / 结果回流 / 列表状态变化
*   二次消费 / 再创作 / 重试 / 复用

复杂模块写深，简单模块写浅；不做形式主义对称。

### 2. 顶层统一，中层半统一，底层不强求对称

*   顶层统一：方便评审、查漏、协作。
*   中层半统一：保证可读性。
*   底层不强求对称：高风险、高状态、高分流模块可以更深。

### 3. 覆盖维度

导图必须覆盖需求中**实际存在**的以下维度；未提供对应要求时，不要凭空补点：

1.  主流程
2.  分支流程
3.  异常流程
4.  权限流程
5.  数据流转
6.  商业化流程
7.  结果回流
8.  二次操作 / 再创作
9.  兼容性 / 性能 / 弱网 / 前后台切换
10. 埋点 / 日志 / 可观测性（仅在需求明确出现时输出）

### 4. 模块归属规则

*   直接由当前模块发起、承接、展示、消费、回流的逻辑，归入当前模块。
*   全局规则若在当前模块触发或被用户感知，应从“当前模块视角”写入当前模块导图。
*   跨页面透传、支付回跳、结果回流、二次编辑等跨模块链路，可以写入当前模块，但必须围绕“当前模块如何发起 / 承接 / 消费 / 感知”来表达。
*   不要把别的模块整套主链路复制进当前模块导图。

### 5. 噪音转译规则

禁止把以下内容原样贴进导图：

*   待定
*   讨论中
*   实现备注
*   研发现状
*   产品纪要原文
*   “详见需求文档”
*   “后续补充”

正确做法：

*   能转译成正式测试点的，先转译再写入。
*   无法确定但确实影响测试设计的，放入“待确认事项”。
*   不影响当前测试设计的噪音，直接忽略。

## 节点质量要求

*   根节点固定为：`{项目名/版本号}`。
*   每个导图的一级业务主题固定且仅有当前模块名。
*   每个文件只承载对应模块内容，不生成额外总图。
*   节点命名统一、中文自然、无空节点、无重复节点。
*   不要只写 ui 位置；页面位置只能作为补充，不能作为主要结构。
*   不允许为了“好看”牺牲真实业务复杂度。
*   不允许为了压缩节点数而删除关键链路、关键状态、关键分流。
*   若节点接近上限，应优先通过“同类归并、链路归并、中文顿号合并、低风险展示点适度归口”来优化结构。

## 测试点导图写法指引

### 一、先看模块声明，再看材料

执行时，先读取用户在请求开头手工声明的模块名单，再按模块逐个建立测试点树。

建议格式：

```text
本次模块为：模块a、模块b、模块c
```

规则：

*   模块名单以用户本次声明为准。
*   声明几个模块，就只生成几个模块导图。
*   不要把设计稿或 prd 中所有页面默认当成导图模块。

### 二、优先写链路，不优先写位置

优先写：

*   入口
*   前置条件
*   输入与参数
*   提交与拦截
*   状态刷新
*   结果展示
*   回流
*   二次操作

避免把“左上角按钮、底部文案、卡片位置”作为主要结构。

### 三、优先写差异点与分流点

重点挖掘：

*   正常 / 异常
*   有权限 / 无权限
*   有数据 / 无数据
*   首次进入 / 再次进入
*   成功 / 失败 / 超时 / 重试
*   会员 / 非会员
*   资源充足 / 资源不足
*   可点击 / 不可点击
*   可回流 / 不可回流

### 四、复杂模块写深，轻模块写浅

不要求所有模块结构对称。复杂模块可以有更多业务主题；承接页、弱逻辑页可以适当归并。

### 五、跨页面链路写法

当某个模块与其他模块存在透传、回流、跳转、回跳关系时，从“当前模块的测试关注点”表达：

*   从哪里进入当前模块
*   当前模块接收了什么
*   当前模块如何消费这份数据
*   后续又把什么状态或结果回流出去

不要把别的模块整套主链路复制过来。

### 六、需求分析简报的正确用法

若用户提供了需求分析简报：

*   可用它帮助识别范围、规则、风险、待确认点
*   不可直接照抄为导图节点
*   不可把“分析结论”当成主输出

主输出仍然是测试点导图。

### 七、无法确认的事项处理

确有歧义时，新增“待确认事项”业务主题，保持精简、正式、可执行，例如：

*   触发优先级待确认
*   扣减时机待确认
*   回跳后是否自动续操作待确认

不要写原始讨论记录。

## xmind 8 输出规范

### 强制格式

必须生成 **xmind 8 / 旧版兼容格式**，禁止生成 zen 格式。

每个 `.xmind` 包内必须包含以下路径：

*   `content.xml`
*   `styles.xml`
*   `meta.xml`
*   `META-INF/manifest.xml`

禁止只输出：

*   `content.json`
*   `metadata.json`

### `content.xml` 约束

根节点必须使用默认命名空间，不允许出现 `ns0:`、`ns1:` 前缀：

```xml
<xmap-content xmlns="urn:xmind:xmap:xmlns:content:2.0" version="2.0">
```

并且必须包含完整的 `sheet`、`topic`、`title`、`children`、`topics` 结构。

### `manifest.xml` 约束

根节点必须为：

```xml
<manifest xmlns="urn:xmind:xmap:xmlns:manifest:1.0">
```

至少包含以下 `file-entry`：

*   `full-path="/" media-type="application/vnd.xmind.workbook"`
*   `full-path="content.xml" media-type="text/xml"`
*   `full-path="styles.xml" media-type="text/xml"`
*   `full-path="meta.xml" media-type="text/xml"`
*   `full-path="META-INF/manifest.xml" media-type="text/xml"`

## 导图内部结构建议

每个模块导图建议按以下抽象结构组织，但不要求机械对称：

```text
项目名/版本号
└── 当前模块名
    ├── 核心链路 / 主流程
    ├── 分支与拦截
    ├── 状态与数据回流
    ├── 商业化 / 权限 / 资源限制
    ├── 结果消费 / 二次操作 / 再创作
    ├── 异常与恢复
    └── 待确认事项（仅在确有必要时保留）
```

要求：

*   一级业务主题只允许 1 个，即当前模块名。
*   当前模块下的业务主题数量、深度、展开方式按真实复杂度决定。
*   节点接近上限时优先做同类归并与链路归并，不删关键覆盖。

## xmind 文件构造要求

虽然本技能不强依赖脚本，但生成文件时必须满足以下结果要求：

1.  产物扩展名为 `.xmind`
2.  本质为 ZIP 包
3.  ZIP 根目录下直接放置：
    *   `content.xml`
    *   `styles.xml`
    *   `meta.xml`
    *   `META-INF/manifest.xml`
4.  禁止多套一层目录
5.  禁止加密、禁止密码保护、禁止出现“password wrong”或“not a valid XMind file”

## 自检要求

输出前必须对每个文件逐项自检，至少覆盖：

1.  `.xmind` 能被 `unzip -t` 正常解压
2.  包内存在：
    *   `content.xml`
    *   `styles.xml`
    *   `meta.xml`
    *   `META-INF/manifest.xml`
3.  `content.xml` 根标签为 `xmap-content`，默认命名空间正确，无 `ns0/ns1`
4.  `manifest.xml` 根标签为 `manifest`，且 `/` 的 `media-type` 为 `application/vnd.xmind.workbook`
5.  当前文件只包含当前模块作为一级业务主题
6.  主题节点数不超过上限
7.  未混入其他模块主链路
8.  未额外生成总图

若任一项失败，必须先修复再输出。

## 输出要求

最终回答中应提供：

1.  每个 `.xmind` 文件的可下载路径
2.  每个文件的主题节点数
3.  自检结果摘要
4.  若存在待确认事项，可额外说明，但不要混入自检结果

## 禁止事项

*   禁止强行让所有模块的二级节点完全相同
*   禁止把需求噪音原样贴入导图
*   禁止只写页面展示，不写状态、数据、权限、回流、商业化
*   禁止把需求分析简报当作主输出
*   禁止生成 zen 格式 `.xmind`
*   禁止跳过校验直接交付
