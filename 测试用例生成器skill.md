# 基于 XMind + PRD + Figma 的全覆盖测试用例编写技能

本技能用于把以下输入整理为一套**可直接执行、可交付、覆盖优先、可追溯、只删除纯冗余**的中文测试用例：

*   XMind 测试点导图
*   PRD / 需求文档
*   Figma 设计稿
*   需求分析简报（可选）
*   补充澄清记录（可选）
*   阶段零范围确认结果（可选）

本技能的核心目标只有两个：

1.  **覆盖一定要全**
2.  **只删除纯冗余用例，不做场景合并压缩**

## 固定执行优先级

1.  **先补齐覆盖**
2.  **再证明覆盖**
3.  **最后只删除完全重复的用例**

禁止为了控制总量而执行以下动作：

*   合并不同场景
*   合并不同异常
*   合并不同回流 / 恢复
*   合并不同边界
*   合并不同正反分支
*   合并不同拦截条件
*   以“综合验证”“闭环验证”“状态流转验证”一条覆盖多个独立高风险场景

允许删除的仅限**纯冗余用例**：

*   风险点相同
*   前置条件相同
*   操作动作相同
*   预期结果相同
*   不删除也不会新增覆盖价值

只要以上五项有任一不同，就不得合并或删除。

## 输入材料优先级

当存在多种输入材料时，按以下优先级解释和使用：

1.  **XMind**：主来源。用于确定模块、链路、测试点层级、测试重点、叶子节点边界。
2.  **PRD / 需求文档**：用于补充业务规则、状态流转、分支条件、边界条件、排除范围。
3.  **Figma**：用于补充页面状态、按钮交互、页面跳转、视觉状态、空态、禁用态、异常态。
4.  **需求分析简报**：用于补充已归纳的规则、缺口、风险、测试重点，但不能替代原始需求。
5.  **补充澄清记录 / 阶段零范围确认结果**：用于覆盖范围收口、处理优先级、确认暂不测项。

禁止编造任何 XMind、PRD、Figma、补充材料中都没有的业务逻辑。

## 输出目标

默认输出为一份可直接交付的中文测试用例表。

默认列头如下：

1.  `用例编号`
2.  `所属产品`
3.  `所属模块/链路`
4.  `测试版本`
5.  `测试标题`
6.  `操作步骤`
7.  `前置条件`
8.  `预期结果`
9.  `实际结果`
10. `附件（备注）`
11. `测试负责人`
12. `时间`

如果用户提供了新的列头、列顺序、模板文件或 Excel 格式要求，必须严格按用户要求输出。

## 强制附加输出

除正式用例表外，必须同步输出以下追溯内容；缺任何一项都不能视为完成：

### 1. 核心测试点覆盖矩阵

至少包含以下列：

*   `核心测试点`
*   `所属模块/链路`
*   `主流程`
*   `关键异常`
*   `关键回流/恢复`
*   `关键边界`
*   `关键正反分支`
*   `关键拦截条件`
*   `对应测试用例编号`
*   `不适用说明`

要求：

*   每个核心测试点都必须逐列判断。
*   适用项必须落到具体用例编号。
*   不适用项必须写原因，不能留空。
*   不允许只写“已覆盖”“见上文”这种不可追溯表述。

### 2. XMind 叶子节点映射表

至少包含以下列：

*   `XMind 原节点路径`
*   `节点类型（展示/交互/规则/状态/异常/结果/全局）`
*   `是否生成独立用例`
*   `对应测试用例编号`
*   `未独立生成原因`

要求：

*   每个 XMind 叶子节点都必须出现一次。
*   每个叶子节点要么映射到至少 1 条用例，要么明确说明为什么不独立成例。
*   禁止出现“模块已覆盖，节点省略”这类宏观替代。

### 3. 待确认事项清单

只能记录：

*   原始需求未定义
*   XMind 与 PRD/Figma 冲突
*   设计未落地但存在明显测试风险

待确认项不得污染正式预期结果。

## 与上游技能的衔接规则

如果用户同时提供以下材料，应按下面方式理解：

*   **需求分析 skill 的产物**：用于理解规则、风险、边界、待确认项，但不能凌驾于原始需求之上。
*   **测试点生成器 skill 的 XMind**：默认视为本技能的主骨架，除非存在明显链路错位或与原始需求冲突。

如果发现上游产物与原始需求冲突：

1.  优先以原始需求为准。
2.  在 `附件（备注）` 中记录冲突来源和处理结论。
3.  不把冲突内容直接扩写成正式用例。

## 适配的项目类型

本技能优先适配以下产品形态：

*   多页面、多步骤流程型产品
*   AI 生成 / 异步任务型产品
*   商业化 / 权益 / 计费 / 订阅型产品
*   存在跨页面参数带入、结果回流、二次创作、状态恢复的产品

这类产品默认优先采用**链路型写法**，但仍需对每个独立风险场景单独成例。

## 核心测试点判定

满足以下任一条件的，视为核心测试点：

*   主链路关键步骤
*   独立业务规则
*   独立状态流转节点
*   独立资格判断 / 拦截条件
*   独立商业化分流点
*   独立参数带入 / 回流点
*   独立恢复 / 重进 / 重试点
*   独立结果消费 / 二次操作点
*   独立展示状态（空态、禁用态、异常态）

对每个核心测试点，必须逐一判断以下 6 个覆盖维度：

*   主流程
*   关键异常流程
*   关键回流 / 恢复流程
*   关键边界
*   关键正反分支
*   关键拦截条件

注意：

*   这是按**每个核心测试点**判断，不是整套用例集平均满足即可。
*   若某一维度不适用，必须在覆盖矩阵中写明原因。
*   不得跳过判断。

## 高风险场景的最小拆分规则

以下场景不得写成 1 条综合用例；必须至少按下列家族拆分检查：

### 1. 支付 / 订阅 / 扣费 / 回补

至少检查：

*   成功
*   失败
*   取消
*   超时
*   回流刷新
*   防重复提交 / 防重复建单
*   权益发放或扣减正确

### 2. AI 任务创建 / 生成 / 重试

至少检查：

*   创建成功
*   创建失败
*   排队中
*   生成中
*   生成成功
*   生成失败
*   超时
*   重试
*   前后台切换恢复
*   强杀重进恢复
*   列表与详情一致

### 3. 权限授权

至少检查：

*   首次授权允许
*   首次授权拒绝
*   拒绝后再次触发
*   去系统设置后回流
*   系统中断或前后台切换恢复

### 4. 下载 / 导出 / 保存 / 分享

至少检查：

*   正常成功
*   权限不足拦截
*   非会员 / 权益不足拦截
*   处理中禁点或防重复
*   失败提示
*   回流后结果一致

### 5. 弹窗 / 引导 / 活动触发

至少检查：

*   触发展示
*   关闭 / 跳过
*   主按钮分流
*   重进后重判
*   配置关闭不触发

### 6. 跨页参数带入 / 结果回流

至少检查：

*   来源参数带入正确
*   回流后页面状态正确
*   历史参数保留或清理正确
*   二次编辑 / 二次生成正确
*   列表 / 详情 / 入口一致

如果输入材料明确了更多独立规则，还要继续拆，不得停在最小集。

## 禁止合并的具体规则

以下情况一律不得合并：

*   成功与失败
*   失败与超时
*   失败与取消
*   首次进入与重进恢复
*   前台路径与后台恢复路径
*   有权限与无权限
*   会员与非会员
*   有数据与无数据
*   配置开启与配置关闭
*   输入合法与输入非法
*   边界内与边界外
*   不同入口触发的同名动作
*   列表验证与详情验证
*   不同拦截条件导致的同一路径中断

## 只允许删除的冗余类型

只有同时满足以下条件，才允许删除其中一条：

1.  覆盖的是同一个核心测试点
2.  覆盖的是同一个覆盖维度
3.  前置条件一致
4.  操作步骤一致
5.  预期结果一致
6.  删除后追溯关系不受影响

否则一律保留。

## 标题必须结构化且可追溯

推荐标题格式：

`xxx（子模块）之xxx（场景）的xxx验证`

例如：

*   `订阅（子模块）之支付成功回流（场景）的闭环验证`
*   `权限（子模块）之首次拒绝后去设置返回（场景）的回流验证`
*   `生成任务（子模块）之超时后重试（场景）的恢复验证`

不要写：

*   `功能验证`
*   `业务验证`
*   `综合验证`
*   `状态流转验证`
*   `测试用例1`

## 用语要求

如果 XMind 节点、PRD 备注或设计稿标注里带有速记、黑话、缩写、研发字段名或临时表述，输出到正式用例时必须先翻译成可执行中文。

规则：

*   正式表格字段一律使用测试团队可直接阅读的中文表达
*   原始速记词可放到 `附件（备注）` 中做溯源
*   不要把字段名、内部口头词、待讨论文案直接放进正式标题

## 字段写法规范

### `所属模块/链路`

写法：

`模块-链路`

例如：

`首页-首启 -> 协议 -> 新人礼 -> 首页`

### `前置条件`

要求短、清楚、可执行。

好的前置条件：

*   应用已安装且可正常启动
*   已登录会员账号
*   当前网络断开
*   已存在可查看作品
*   当前账号积分不足
*   当前存在生成中任务

### `操作步骤`

要求：

*   每一步都必须可操作
*   每一步都必须可观察
*   一个独立风险场景单独成例
*   不要使用“进行相关操作”“检查页面是否正常”这种空话

### `预期结果`

必须写成 QA 可检查结果，优先写：

*   页面停留在正确路由
*   按钮可点 / 不可点状态正确
*   列表、详情、角标、余额、权益展示刷新正确
*   状态流转正确
*   无重复建单
*   无脏数据
*   参数带入正确
*   任务结果正确回流
*   扣减或回补正确
*   拦截后提示、停留页、按钮状态正确
*   重进、回流、恢复后的状态与动作一致

避免写：

*   `符合预期`
*   `系统正常`
*   `逻辑正确`

### `附件（备注）`

用于记录：

*   原始 XMind 速记词
*   对应设计稿页面名
*   对应需求章节
*   冲突处理说明
*   待确认事项编号

不要把主验证逻辑塞进备注里。

## 工作流程

### 第一步：梳理输入材料

先读：

*   XMind
*   PRD / 需求文档
*   Figma
*   需求分析简报（如有）
*   补充澄清记录（如有）
*   范围确认结果（如有）

然后建立：

*   模块
*   链路
*   子模块
*   场景
*   叶子节点
*   全局规则
*   排除范围

### 第二步：抽取核心测试点

对每个 XMind 叶子节点和每条明确业务规则，判断其是否为核心测试点。

注意：

*   不得因为“已在相邻节点覆盖”而跳过。
*   不得因为“已经有总用例”而跳过。
*   不得先做归一化或场景合并。

### 第三步：逐点做 6 维覆盖检查

对每个核心测试点逐一检查：

*   主流程
*   关键异常
*   关键回流 / 恢复
*   关键边界
*   关键正反分支
*   关键拦截条件

规则：

*   适用就写成独立用例。
*   不适用就写入覆盖矩阵中的不适用原因。
*   不得用“一条综合用例”替代多个独立适用维度。

### 第四步：生成正式用例

每个独立风险场景生成独立用例，明确：

*   `所属模块/链路`
*   结构化标题
*   可执行前置条件
*   可操作步骤
*   可判定预期结果

### 第五步：删除纯冗余

只允许删除完全重复的用例，不允许任何场景合并。

### 第六步：输出追溯附件

必须输出：

*   正式测试用例表
*   核心测试点覆盖矩阵
*   XMind 叶子节点映射表
*   待确认事项清单

### 第七步：交付前硬校验

交付前逐条检查以下门禁。只要有一条不满足，就继续补写，不能交付：

1.  每个核心测试点都完成了 6 维判断。
2.  每个适用维度都有具体用例编号。
3.  每个 XMind 叶子节点都有映射去向。
4.  高风险家族没有被压缩成综合用例。
5.  不存在“模块已覆盖”“同类省略”“见上文”这类不可追溯表述。
6.  正式预期中没有待确认逻辑。
7.  删除动作只发生在纯冗余用例上。

## 明确禁止生成的内容

除非用户明确要求，否则不要生成：

*   埋点
*   日志
*   审计
*   观测

同时避免使用空泛表达：

*   `业务入口`
*   `业务动作`
*   `业务数据`

优先替换成：

*   `页面入口`
*   `页面展示`
*   `页面状态`
*   `接口返回`
*   `任务状态`
*   `会员权益`
*   `积分余额`
*   `作品列表`
*   `结果详情`

## 最终产出态度

默认要做到：

*   覆盖优先
*   不漏点
*   可追溯
*   可回归
*   可直接执行
*   不做场景合并
*   只删纯冗余

正确结果不是“条数最少”，而是：

**在覆盖充分、逐点可追溯、风险不被压缩的前提下，输出一套完整且可执行的测试用例集。**
