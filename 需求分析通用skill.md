# PRD 需求分析简报

## 概述

使用本 skill 将 PRD 或需求文档转化为一份**面向测试设计的需求分析与测试理解简报**（Markdown）。

目标不是复述 PRD，而是：

*   明确被测主体与测试边界；
*   提取稳定、可测试、可追溯的规则；
*   主动识别需求缺口、歧义与风险；
*   为后续测试点、测试用例、三层交叉验证提供上游输入。

本 skill 默认只以**当前输入的 PRD / 需求文档**作为事实来源。
除非用户在当前对话中明确补充额外需求文本，否则不要引入其他资料，也不要自行补全产品行为。
若文档未定义，必须记录为“需求缺口 / 待确认项 / 假设”，不得伪造确定性结论。

## 输出原则

始终输出**一份 Markdown 格式的《需求分析与测试理解简报》**。
这份简报必须能够直接衔接：

*   测试点拆解；
*   测试用例编写；
*   三层交叉验证报告。

因此，输出必须满足以下要求：

*   章节名称稳定；
*   规则编号稳定；
*   缺口编号稳定；
*   表格字段尽量固定；
*   重要行为必须能追溯到规则、缺口、标准或风险之一。

## 适用场景

当需求文档具有以下任一特征时，优先使用本 skill：

*   存在多页面、多入口或多步骤流程；
*   存在跨页参数带入、默认值继承、预填充；
*   存在异步任务、状态流转、重试、超时、失败回滚；
*   存在后台动态配置、功能开关、模板配置、排序配置；
*   存在系统权限、网络、相机、相册、分享、支付等外部依赖；
*   存在会员、积分、订阅、价格、退款、恢复购买、权益过期等规则；
*   存在边界复杂、歧义较多、需要先做测试向需求澄清的产品需求。

## 工作流程

按以下顺序执行分析。

### 第 1 步：登记输入来源

先在简报中记录所有输入的 PRD / 需求文档来源。
对每个来源，记录：

*   文档名称；
*   可见版本号（若有）；
*   是否为主文档；
*   是否为补充文档。

如果用户只提供了一份文档，应明确写出“当前分析仅基于该单一来源”。

### 第 2 步：声明生效的分析规则

输出一个简短但可执行的“生效分析规则”章节，至少说明：

*   来源优先级规则；
*   歧义处理规则；
*   追溯规则；
*   禁止脑补规则；
*   条件章节启用规则。

默认使用以下规则，除非用户另有要求：

*   PRD / 需求文档为主事实来源；
*   文档未定义时，记录缺口，不用猜测填补；
*   每条可测试规则必须分配稳定编号；
*   所有覆盖表、缺口表、规则表、标准表应复用同一套编号体系；
*   仅在确实适用时，才启用复杂产品的扩展分析章节。

### 第 3 步：界定被测主体范围（SUT）

将需求内容划分为三个区域：

| 区域     | 含义            | 测试处理          |
| ------ | ------------- | ------------- |
| Zone 1 | 被测主体直接实现的行为   | 完整分析，后续完整设计测试 |
| Zone 2 | 与被测主体交互的上下游区域 | 保留背景与接口边界分析   |
| Zone 3 | 本次不纳入深测的外部区域  | 仅登记，不展开测试设计   |

如果用户已经提供“测试范围确认结论”，直接引用。
如果没有，则基于当前 PRD 推断一个**暂定范围**，并明确标记：

*   ⚠️ 当前为基于 PRD 推断的暂定范围，待后续测试范围确认。

不要把“推断得到的范围”写成“已确认范围”。

### 第 4 步：建立文档结构覆盖追踪

按文档结构逐章审阅，建立“文档结构覆盖追踪表”。
目的不是做目录摘录，而是证明分析覆盖了整份需求，而不是只看主链路。

对每个一级/二级章节，记录：

*   覆盖状态；
*   Zone 映射；
*   提取的规则编号；
*   备注。

覆盖状态仅允许使用：

*   ✅ 已覆盖
*   ⚠️ 部分覆盖
*   ❌ 未覆盖

### 第 5 步：识别需求缺口

主动寻找影响测试设计的需求不完整之处。
至少检查以下类型：

*   错误处理未定义；
*   行为规则需靠推断；
*   数值/字数/范围/频次边界缺失；
*   状态迁移规则缺失；
*   跨页参数继承规则缺失；
*   异步生命周期缺失；
*   并发 / 竞态 / 去重规则缺失；
*   文档内前后表述矛盾；
*   存在商业化逻辑但权益、价格、退款规则不全；
*   存在动态配置但降级与兜底规则不明；
*   存在系统能力依赖但权限拒绝或能力不可用路径不明。

每个缺口必须包含：

*   缺口编号；
*   缺口类型；
*   严重程度；
*   缺口描述；
*   对测试的影响；
*   建议澄清方向。

严重程度使用：高 / 中 / 低。

### 第 6 步：提取关键规则

将需求拆解为具备稳定编号的关键规则。
使用以下前缀：

| 规则类型 | 编号前缀   | 说明                 |
| ---- | ------ | ------------------ |
| 数据规则 | DR-xxx | 数据格式、字段、取值、边界、输入限制 |
| 行为规则 | BR-xxx | 业务行为、交互行为、功能分支     |
| 流程规则 | PR-xxx | 步骤顺序、状态流、页面流、任务流   |
| 约束规则 | CR-xxx | UI、布局、技术限制、能力约束    |
| 安全规则 | SR-xxx | 权限、隐私、授权、数据保护      |

规则要求：

*   尽量原子化；
*   必须可测试；
*   尽量避免一条规则覆盖多个不同判断点；
*   用“规则陈述句”表述，不写成模糊总结。

优先拆成多条可测规则，不要写成过于宽泛的大规则。

### 第 7 步：按需启用复杂产品扩展分析章节

以下章节仅在 PRD 具备对应特征时启用；不适用时可省略，或标记为“不适用”。

#### 7a. 状态机与状态迁移分析

当需求存在用户状态、任务状态、订阅状态、支付状态、审核状态、权益状态等“状态驱动行为”时启用。

至少分析：

*   状态集合；
*   进入条件；
*   离开条件；
*   触发事件；
*   禁止迁移；
*   文档未定义的状态迁移。

#### 7b. 页面跳转与参数继承矩阵

当需求存在多入口同功能、页面预填、默认值继承、跨页带参时启用。

至少分析：

*   来源页面 / 来源上下文；
*   触发动作；
*   目标页面；
*   带入参数；
*   默认值规则；
*   带入后可否修改；
*   返回预期；
*   风险备注。

#### 7c. 异步任务生命周期分析

当需求中存在提交、排队、生成、同步、后台处理、失败、超时、重试等异步行为时启用。

至少分析：

*   提交前校验；
*   任务创建时机；
*   处理中；
*   成功；
*   失败；
*   超时；
*   重试；
*   重复提交风险；
*   若涉及消耗，需分析扣费或回退时机。

#### 7d. 动态配置与降级规则

当模块、卡片、模板、价格、标签、文案、排序、开关、能力可见性等由后台配置驱动时启用。

至少分析：

*   配置项；
*   是否必填；
*   正常行为；
*   客户端兜底；
*   空数据行为；
*   非法值行为；
*   加载失败行为；
*   开关关闭行为。

#### 7e. 权益 / 计费 / 账本规则分析

当需求存在会员、积分、订阅、价格、补差、退款、恢复购买、权益过期、资产优先级等规则时启用。

至少分析：

*   资产 / 权益类型；
*   获得方式；
*   有效期；
*   消耗顺序；
*   余额不足行为；
*   升降级规则；
*   退款影响；
*   恢复规则；
*   文档中明确提到的跨账号 / 跨设备口径；
*   未闭合的账本规则缺口。

#### 7f. 外部依赖与系统能力分析

当需求依赖相机、相册、网络、分享、支付 SDK、通知、定位、存储等系统能力或外部服务时启用。

至少分析：

*   依赖项；
*   触发时机；
*   成功路径；
*   拒绝 / 失败路径；
*   重试或替代路径；
*   用户可见兜底方式。

## 各章节要求

### 来源文档

列出输入文档名称与可见版本信息。
若文档未体现显式版本号，应明确写“文档未标注版本”。

### 0a. 生效的分析规则

保持简洁、可执行。
重点写清分析优先级、歧义处理与追溯规则，不要写空泛理论。

### 0b. 被测主体范围界定（SUT Scope Definition）

固定输出：

*   被测主体名称；
*   Zone 划分表；
*   范围确认状态。

若范围并非用户确认，而是基于 PRD 推断，必须显式写出：

*   ⚠️ 当前为基于 PRD 推断的暂定范围，待后续测试范围确认。

### 1. 文档结构覆盖追踪

使用表格，建议列如下：

*
#
*   一级章节
*   二级章节
*   覆盖状态
*   Zone
*   提取的规则
*   备注

必须体现对完整 PRD 结构的覆盖，而不是仅覆盖部分功能点。

### 2. 需求缺口清单

使用结构化表格或编号列表。
每条缺口都应能直接用于后续与产品 / 开发沟通澄清。

### 3. 关键规则

按规则类型分组提取。
每条规则都要简洁、可测、唯一。

推荐表头：

| 规则编号 | 规则类型 | 规则内容 | 来源位置 |
| ---- | ---- | ---- | ---- |

### 3a. 状态机与状态迁移分析

仅在适用时输出。

推荐表头：

| 对象 | 状态 | 进入条件 | 离开条件 | 关键转移 | 缺口/风险 |
| -- | -- | ---- | ---- | ---- | ----- |

### 3b. 页面跳转与参数继承矩阵

仅在适用时输出。

推荐表头：

| 来源页面/上下文 | 触发动作 | 目标页面 | 带入参数 | 可修改性 | 返回预期 | 风险 |
| -------- | ---- | ---- | ---- | ---- | ---- | -- |

### 3c. 异步任务生命周期分析

仅在适用时输出。

推荐表头：

| 阶段 | 触发条件 | 用户可见表现 | 数据/状态变化 | 异常点 | 备注 |
| -- | ---- | ------ | ------- | --- | -- |

### 3d. 动态配置与降级规则

仅在适用时输出。

推荐表头：

| 配置项 | 来源 | 必填性 | 正常行为 | 缺省/异常降级 | 风险 |
| --- | -- | --- | ---- | ------- | -- |

### 3e. 权益 / 计费 / 账本规则分析

仅在适用时输出。

推荐表头：

| 规则编号 | 资产/权益类型 | 获得方式 | 有效期 | 消耗/限制规则 | 退款/恢复影响 | 风险 |
| ---- | ------- | ---- | --- | ------- | ------- | -- |

### 3f. 外部依赖与系统能力分析

仅在适用时输出。

推荐表头：

| 依赖项 | 触发点 | 成功路径 | 失败/拒绝路径 | 重试或替代路径 | 风险 |
| --- | --- | ---- | ------- | ------- | -- |

### 4. 测试标准

这一章必须能直接为后续测试用例设计提供判定依据，不要写成空泛口号。

优先从以下维度组织：

*   正确性标准；
*   状态流转标准；
*   异常处理标准；
*   权益 / 计费标准（若适用）；
*   性能 / 可靠性标准（仅当 PRD 有定义时输出）；
*   可观测性标准（仅当 PRD 有定义时输出）。

推荐表头：

| 操作/场景 | 正确性标准 | 状态流转标准 | 异常处理标准 | 其他标准 | 备注 |
| ----- | ----- | ------ | ------ | ---- | -- |

### 5. 测试理解简报

这一章负责“综合理解”，不是重复前文内容。
应至少包含以下子章节：

1.  系统流程概述；
2.  核心规则 Top 5；
3.  关键风险；
4.  测试重点分配。

对于“关键风险”，至少说明：

*   风险领域；
*   风险描述；
*   潜在影响；
*   风险等级；
*   建议测试重点。

“测试重点分配”使用 P0 / P1 / P2 标记。

### 6. 背景知识

只保留有助于后续测试理解依赖关系的背景，不要写泛泛的产品介绍。
例如：

*   上下游关系；
*   PRD 中明确出现的外部系统；
*   运行时依赖或环境假设；
*   文档中明确暗示的用户角色前提。

### 7. 分析结论

始终包含：

*   需求完整度评估；
*   可测试性评估；
*   待澄清问题 / 跟进项（按优先级排序）；
*   建议测试策略。

测试策略标签应与文档特点相匹配，例如：

*   风险驱动；
*   功能驱动；
*   状态驱动；
*   交互驱动；
*   计费驱动。

允许组合多个策略，但必须有依据。

### 补充知识集成

仅当用户后续补充了口头澄清、会议信息、补充规则时再追加本章节。
对每条补充信息，记录：

*   来源与日期；
*   新增或更新的规则编号；
*   风险变化；
*   测试重点调整。

## 面向下游测试产物的追溯规则

这份简报必须天然支持后续测试设计，因此：

*   规则编号应稳定；
*   缺口编号应稳定；
*   章节名应稳定；
*   同一轮分析中不得静默改号；
*   重要可测行为至少落入“规则 / 缺口 / 标准 / 风险”之一。

编写时要始终考虑后续链路：

*   **测试点** 应能追溯到规则、缺口、风险；
*   **测试用例** 应能引用规则编号与场景编号；
*   **三层交叉验证报告** 应能执行“规则 → 用例、用例 → 规则、标准/风险 → 用例”的反查。

## 风格要求

*   默认使用中文输出，除非用户明确要求其他语言；
*   简洁但完整；
*   追溯性强的章节优先使用表格；
*   明确区分事实、推断、假设、待确认项；
*   当 PRD 未支撑某个判断时，明确写“PRD 未定义”或“需确认”；
*   不得把当前项目之外的历史经验伪装成本文档结论；
*   不要写入与当前输入文档无关的项目细节。

## 输出前自检

在输出简报前，检查以下事项：

1.  是否明确了 SUT 边界；
2.  是否覆盖了主要 PRD 结构；
3.  是否为主要行为建立了规则或缺口；
4.  复杂产品特征是否启用了对应扩展章节；
5.  结果是否可直接衔接测试点与测试用例设计；
6.  是否把无依据的推断误写成事实。
